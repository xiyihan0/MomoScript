## ADDED Requirements

### Requirement: Parser implementation preserves syntax and semantic boundaries

下一版 DSL parser SHALL 产出可用于 IDE 和后续 semantic lowering 的 syntax AST，而不是在 syntax 阶段直接执行资源解析或渲染展开。

#### Scenario: Syntax nodes preserve source ranges

- GIVEN parser 读取 statement、directive block、inline marker 或 patch
- WHEN 它产出 syntax AST
- THEN each syntax node SHOULD preserve a UTF-8 byte-based source range
- AND diagnostics SHOULD expose user-facing line and column positions derived from that range

#### Scenario: Directive blocks are parsed generically

- GIVEN 作者写出 `@actor ... @end` 或 `@asset ... @end`
- WHEN syntax parser 读取该 block
- THEN parser SHOULD produce a generic directive block syntax node containing directive name, head arguments, fields, body items, and source range
- AND `@actor` / `@asset` field validation SHOULD happen in semantic lowering rather than the syntax phase

#### Scenario: Nested directive blocks are not supported in the first revision

- GIVEN parser 正在读取一个 directive block
- WHEN block 内出现新的明确顶层 directive block 起始
- THEN parser MUST report a diagnostic for unsupported nested block syntax
- AND MUST NOT silently reinterpret the inner block as an independent valid top-level declaration

#### Scenario: Recoverable parse errors remain visible

- GIVEN parser encounters a malformed block or malformed node
- WHEN it recovers to continue parsing later top-level nodes
- THEN parser MUST preserve the malformed source region as an error node or equivalent diagnostic range
- AND strict compilation MUST fail while error diagnostics remain present

### Requirement: Generated Typst diagnostics map back to MMT origins

下一版 emitter SHALL 在生成 Typst 时维护 source map，以支持 Typst diagnostic 反查到 MMT 源范围。

#### Scenario: Typst diagnostics use generated byte ranges

- GIVEN emitter 生成 `.typ` 文本并交给 Typst 0.15 编译
- WHEN Typst returns a `SourceDiagnostic`
- THEN compiler SHOULD map `SourceDiagnostic.span` to a generated Typst UTF-8 byte range
- AND SHOULD use the generated source map to find the closest MMT origin range

#### Scenario: Source map is built from emitted chunks

- GIVEN emitter writes template wrappers, escaped text, direct Typst body, inline macro expansion, or patch arguments
- WHEN it appends text to the generated Typst document
- THEN it SHOULD associate each emitted chunk with an origin
- AND generated wrapper chunks SHOULD retain a parent origin such as the owning statement, marker, or directive

#### Scenario: Zero-length Typst ranges are supported

- GIVEN Typst reports a diagnostic at an insertion point with an empty generated range
- WHEN compiler looks up that range in the source map
- THEN lookup SHOULD support zero-length range matching
- AND SHOULD return the most specific adjacent or containing MMT origin when available

#### Scenario: Resource patch errors map to patch origins

- GIVEN `[:...:](...)` expands into a generated Typst resource call
- WHEN Typst reports an error inside the generated arguments that came from the marker suffix patch
- THEN compiler SHOULD report the MMT origin as the resource patch range
- AND SHOULD NOT collapse the diagnostic to the whole statement unless no more specific origin exists

#### Scenario: Pipeline diagnostics keep phase identity

- GIVEN an error happens before final Typst compilation
- WHEN the error occurs in syntax parsing, semantic lowering, resource resolution, or materialization
- THEN compiler SHOULD report that phase directly
- AND SHOULD use the original MMT node origin instead of creating a synthetic Typst diagnostic

#### Scenario: Debug comments do not replace source maps

- GIVEN emitter optionally inserts debug comments into generated Typst output
- WHEN compiler maps diagnostics back to MMT source
- THEN machine mapping MUST rely on source map data
- AND MUST NOT depend on parsing debug comments from generated Typst text
